## 문제 정의

수영을 시작하면서 느낀 불편을 그대로 출발점으로 삼았다.

- 기록은 A앱, 동호회 정보는 B앱, 대회 기록은 C앱 — **매번 앱을 갈아타야 했다.**
- 자세 교정은 비싼 1:1 레슨이 아니면 객관적 피드백을 받기 어려웠다.
- 초보자는 "오늘 뭘 연습해야 하는지" 스스로 설계하지 못한다.

→ **분석·기록·훈련 설계·커뮤니티를 한 곳에서** 해결하는 것을 목표로 잡았다.

## 핵심 기능

### AI 영법 분석 *(내부 테스트 중)*
업로드한 수영 영상에서 관절 좌표를 추출하고, 시간에 따른 움직임 패턴으로 영법을 판별한 뒤 자세 교정 포인트를 제시한다.
- 영법 분류 정확도 **94.4%**, 관절 검출 정확도 **81.5%**
- 상세 동작 원리 → [AI 영법 분석 기술 문서 링크 자리]

### 훈련 관리
- **훈련 일지** — 영법·거리·강도 기록, 캘린더 뷰, 월간 목표 달성률
- **훈련 플랜** — 목적별 4주 플랜, 25m/50m 수영장 선택, 세트별 사이클 입력, 드릴/대시 구성 선택, 드래그앤드롭 커스텀 빌더
- **월간 리포트** — 영법별 분포·주차별 추이·성장률, 이미지 공유

### 커뮤니티 & 동기부여
- **커뮤니티** — 게시판·댓글·좋아요·신고·마크다운 에디터·인기글
- **수영 챌린지** — 거리/연속출석 챌린지, 훈련 일지 연동 자동 달성률
- **코치 연동** — 초대 코드로 코치-수강생 연결, 피드백·플랜 전달

### 정보·도움
수영장 찾기(지도), 드릴 가이드, 부상 예방, 용어 사전, 장비 가이드, 영상 큐레이션, FAQ

## 아키텍처
[브라우저] → [Caddy (HTTPS 리버스 프록시)]
├─ FastAPI (API 서버)
│     ├─ PostgreSQL (데이터)
│     ├─ Redis (캐시/세션)
│     ├─ MinIO (이미지/영상 저장)
│     └─ Celery Worker (영상 분석 비동기 처리)
└─ Flowise + Google Gemini (AI 챗봇)

- **7개 컨테이너** Docker Compose 오케스트레이션 (총 RAM ~1.45GB)
- 영상 분석은 Celery로 **비동기 처리** → API 응답 블로킹 방지
- AI 챗봇은 Flowise 경유로 Gemini 호출 (무료 티어 활용)

## 엔지니어링에서 신경 쓴 것

기능 개수보다, 만드는 과정에서의 판단을 남기려 했다.

- **테스트 자동화** — 기능 추가마다 Playwright 테스트를 함께 작성, 96개 유지. 회귀 버그를 배포 전에 잡는다.
- **데이터 품질 우선** — 영법 분류 정확도 40%→94.4% 개선의 핵심은 모델 교체가 아니라 **입력 전처리(CLAHE)와 좌표 안정화(Kalman), 학습 데이터 확충**이었다.
- **비용 의식적 설계** — 배포처를 AWS 프리티어(1GB, RAM 부족) 대신 Hetzner로 분석, 무료 범위 내 운영 구조 설계.
- **디자인 시스템화** — 색·폰트·간격을 토큰으로 추출(DESIGN_TOKENS.md), 17개 페이지 일관성 확보.

## 회고

이 프로젝트는 혼자 진행했지만, 모든 코드를 직접 타이핑하지는 않았다.
대신 **무엇을 만들지 구조를 설계하고, AI(Claude Code)로 구현한 뒤,
결과를 직접 검토·검증하는** 방식으로 1인 생산성을 끌어올렸다.

- **설계는 내 몫이었다** — 데이터 파이프라인 구조, DB 스키마, 기능 우선순위,
  배포 환경 비교(AWS vs Hetzner)는 직접 판단했다.
- **검증도 내 몫이었다** — 96개 자동 테스트를 기준으로, 구현된 코드가
  의도대로 동작하는지 매번 확인했다. 테스트가 통과해야 다음으로 넘어갔다.
- **비효율은 자동화했다** — 반복되는 ML 학습·테스트·배포 과정을
  스크립트 하나로 묶어(run_pipeline.bat: 수집→특징추출→검증→학습→평가→배포)
  수작업을 제거했다.
- AI는 구현 속도를 높이는 도구였지, 판단을 대신하지 않았다.

가장 오래 붙들었던 건 **AI 영법 분석의 정확도**였다.
분류 정확도가 40%(거의 무작위)에서 멈춰 있었고, 특정 영법(접영)은 0%였다.
처음엔 모델을 바꾸면 해결될 거라 생각했지만, 아니었다.
- 진짜 원인은 **입력 데이터의 품질**이었다. 물속 영상은 반사·물보라 때문에
  관절 검출이 출렁였고, 그 노이즈가 그대로 분류 정확도를 끌어내렸다.
- CLAHE 전처리로 대비를 높이고, 칼만 필터로 좌표를 안정화하고,
  부족했던 영법의 학습 데이터를 확충하자 분류 정확도가 **94.4%**,
  관절 검출이 **81.5%**까지 올라갔다.

여기서 배운 건 분명했다 — **모델보다 데이터가 먼저다.**
성능이 안 나올 때 알고리즘을 의심하기 전에, 입력으로 들어가는 데이터가
깨끗한지부터 보게 됐다. 같은 함정을 여러 번 겪고 나서야 체득한 감각이다.
